# x86_64 — the second target: scope and design ledger

Status: design ratified in discussion 2026-07-06; this file is the
record. The model instantiates the target-model contract of
docs/LOWERING.md §8; the front/back split (§6ae) is what makes a second
target a bounded exercise — tools/low/{shape,schema,lin,doc}.shard
import unchanged, proof.shard forks, a new back end plays lowergen's
role. This document pins scope and the abstraction ledger BEFORE the
model exists; probe findings and slice records get appended as they
land.

## 1. Why x86_64, and why now

The wasm pilot is complete through Ring 0 + the loop-fragment family +
the front/back split; the §8 contract is distilled from everything it
needed. x86_64 is the target that tests the contract's genericity where
wasm was kind: no typed frames for free, no structured control in the
machine, flat shared memory.

Straight to x86_64 — **no 32-bit x86 model, ever**. The easiest-first
rung 32-bit would have occupied was wasm's explicit job, and 32-bit is
worse on every axis that matters here: cdecl passes args on the stack
(forcing the stack question immediately), 8 GPRs vs 16, and no native
differential without compat modes. x86_64 SysV passes the first six
args in registers, which composes directly with the frame design below.

The dev box IS the target: the engine-differential contract item (§8.6)
is satisfied by **real silicon** — a harness that writes encoded bytes
into an executable page and calls them with SysV register args (the
x86 analog of wasm_diff.mjs/V8, and the realest reality gate
available).

## 2. The governing principle: performance parity

User-set bar: the architecture must place no blocker to eventual parity
with C+gcc / Rust+LLVM workloads. Parity is not today's deliverable;
today's obligation is that **every abstraction in the model is
classified** into one of three bins:

- **permanent and perf-neutral** — abstractions matching the contract
  gcc/LLVM themselves offer their users; never removed;
- **training wheels with a named removal path** — v1 omissions whose
  later addition is grammar/convention GROWTH, touching no existing
  cert;
- **rework bombs** — decisions whose later migration would re-template
  the proof corpus. These get decided NOW, in favor of real x86_64
  usage.

One standing perf note in the pipeline's favor: PRE discipline
discharges bounds by proof, so safety costs zero runtime checks — the
certified pipeline is AHEAD of parity on that axis, not behind.

## 3. Permanent abstractions (perf-neutral)

1. **Structured (reducible) control.** The model executes a program
   VALUE — an AST with regions/loops/ifs — never bytes from memory.
   Compilers compile reducible CFGs; performance never comes from
   irreducible flow. The real perf patterns this seems to exclude all
   have structured representations proven by wasm itself and enter as
   grammar growth: jump tables (BrTable analog), indirect calls
   (call-through-table), tail calls.
2. **Code is not data (Harvard split in the model).** Mid-instruction
   jumps, overlapping decode, self-modifying code are INEXPRESSIBLE —
   excluded by the grammar, not policed by a checker. The flattening
   encoder (structured control → labels → relative jumps → bytes) is a
   differential-covered trust leaf exactly like wasm's encoder.
3. **Private control stack.** Return addresses and the call/ret
   mechanism stay model-level and are never addressable. Zero perf
   cost — no legitimate C/Rust pattern reads a return address (UB
   there too); LLVM offers the same guarantee shape.

## 4. Commit-now decisions (rework bombs if deferred)

1. **The real 16-GPR register file, in the model, day one.** Register
   allocation is the EMITTER's job at generation time, so the generated
   cert speaks about the ALLOCATED program. This exploits the
   pipeline's own LCF discipline: the generator is untrusted, the
   output is proof-checked, so allocation correctness is swallowed by
   the per-program proof — no verified-regalloc arc ever, a near-1:1
   encoder (small trust leaf), and performance-relevant placement
   visible in the certified layer. The rejected alternative (unbounded
   virtual slots, encoder allocates — the wasm-shaped comfortable
   choice) parks regalloc in the uncertified leaf AND re-templates
   every proof at the eventual migration. Proof templates spell
   rdi/rax instead of local indices; comparable complexity to wasm's
   locals arithmetic.
2. **SysV calling convention from day one.** Args in
   rdi/rsi/rdx/rcx/r8/r9, return in rax. The caller-/callee-saved
   split enters the standard cert form as a **register clobber set** —
   the register analog of memory footprints: the call-composition
   keystone carries "callee preserves these registers" the way
   footprint companions carry "callee doesn't touch these bytes."
   Starting wasm-style (calls preserve everything) would bake a
   fiction into every consumer proof.
3. **The control/data stack split.** Control stack private forever
   (§3.3). The DATA stack — spill slots, stack-allocated buffers,
   out-params — must eventually be ADDRESSABLE Mem, because heap-free
   stack allocation is the heart of the embedded-Rust patterns pinned
   as first-class. It is a convention, not a mechanism: a
   stack-pointer register + footprint discipline over a Mem region
   (the §7.3 footprint-companion promotion), added when a consumer
   forces it. LLVM→wasm makes exactly this split (protected call
   stack + shadow stack in linear memory) at near-native performance.
   The pin here is negative: frames are NOT unaddressable-forever.
4. **Value domain [0, 2^64), wrap64 collapse family.** The
   rep-collapse lemma family (§8.4) at modulus 2^64. Engine
   consequence probed before the model (probe 0a below): 64-bit
   constants and 2^128-scale ground products exercise the compiled
   chain's BigInt path where the wasm pilot never left i63 territory.

## 5. Training wheels (named removal paths, all additive)

- **Byte-width memory ops only** → wide loads/stores (mov
  byte/word/dword/qword). std/mem's `mem_read` width-first surface
  anticipates this; growth = new ops + collapse lemmas, no existing
  cert touched.
- **Register-only operands** → real addressing modes. The operand
  position in the instruction grammar is a DATATYPE from day one; v1
  populates only the register arm; [base + scale·index + disp] arrives
  as new arms + emitter selection, not a grammar rework.
- **FLAGS stay out of the model.** Comparisons fuse into structured
  conditions (the If carries the comparison; encoder emits cmp+jcc) —
  the cond_op shape. Named perf exceptions enter later as COMPOUND ops
  when consumers arrive: x86 div yielding (quot, rem) in one
  instruction, widening 64×64→128 mul, adc for bignum chains.
- **One canonical encoding per instruction.** The encoder is total on
  the grammar, mechanical REX+ModRM, zero optimization. Encoding
  selection, alignment, scheduling, uop fusion are semantics-neutral
  encoder/micro-arch concerns — differential-covered, never certified.
- **No spills in v1** (current fragments fit in 16 registers); spill
  vocabulary = data-stack slots per §4.3 when something forces it.
- **Optimization generally**: the pipeline is already on the
  translation-validation side of the CompCert tradeoff — inlining is
  ratified as "a general certified transform" and is the template for
  later scheduling-class rewrites.

## 6. SIMD (eventually; extension points only, nothing built)

Machine state carries register CLASSES from day one: the GPR file now,
an XMM/YMM file absent-or-empty until used — the precaution is that
state shape must not bake in "registers = one Int list." Op tables are
the extension mechanism; the wasm lesson is pinned: the Bop vocabulary
is FIVE tables wide (semantics / encoder / cert readback / render /
emitter conditions) — extend in step or drift. Lane representation
(List Int vs wide Int) genuinely deferred.

## 7. The §8 contract instantiation checklist

What "add x86_64" means, item by item (LOWERING.md §8):

1. Fuel big-step, additive slack: same discipline; call entry burns
   fuel; loops re-enter at fuel−1 (structured control makes the wasm
   law transplant directly — probe A verifies).
2. Named SCC stop points: eval_call/eval_loop/eval_seq analogs in the
   x86 evaluator's recursion split.
3. Call-composition keystone: call_bridge analog, EXTENDED with the
   register clobber set (§4.2) — the one place the x86 keystone is
   richer than wasm's.
4. Rep-collapse family: wrap64_id + friends (probe 0a stresses the
   constants).
5. Literal-spelling discipline: unchanged (ENC opacity law applies to
   any Nat/register-name enc fns riding into machine state).
6. Encoder + engine differential: flattening encoder + the
   mmap-and-call silicon harness (probe 0b proves the leg exists).
   Trap leg: where the model says None, the harness must observe a
   signal (SIGFPE/SIGSEGV) or refuse-by-construction — open question,
   settle at probe B.
7. Memory denotation: std/mem's observational substrate UNCHANGED —
   same Mem, same mask-on-read, same framed arbitrary-m0 statements.
   Flat shared memory means footprint companions (§7.3) promote from
   QoL toward mandatory as multi-module x86 builds arrive; let a real
   consumer force the timing.

## 8. Probe plan (pre-model gates first, then the go/no-go pair)

- **Probe 0a — BigInt/i63 range** (examples/w64_probe.shard): 2^64
  constants in farkas certificates (wrap64_id = wrap32_id transplanted),
  2^128 ground products, 64-bit shifts, through BOTH engines
  (bin/shard_check claims; bin/shard_eval run-mode — the compiled
  chain's documented i63 mul/shift traps are the thing measured).
  **RUN 2026-07-06, findings:** the Rust interpreter (the semantics
  authority) is fully 64-bit-clean — run-mode ground facts all OK and
  the interpreted checker (`eval run kernel/check.shard`) passes 4/4
  including wrap64_id's 2^64 farkas coefficients and the 2^128 ground
  product. The COMPILED chain traps loudly ("rt trap: mul overflow")
  on any literal above the i63 line — including PARSING a 2^64
  decimal literal (mul-by-10 accumulation); 2^62−1 passes, boundary
  exact. Shift prim (rust_bootstrap/src/prim.rs shift_amount): counts
  0..63 compute, ≥64 leave the prim stuck — exactly x86_64's own
  masked-shift domain, so no model impact. CONSEQUENCE: x86_64 model
  work runs on the interpreted engines until the compiled chain's
  BigInt debt is paid; the fast KERNEL-gate path (bin/shard_check) is
  unusable for files carrying 2^64-scale literals, loud never silent.
- **Probe 0b — mmap+call feasibility**: hand-assembled add-function
  bytes into an executable page (W^X-safe mprotect flip), called with
  SysV args, wrap64 edge checked on silicon. Proves the differential
  leg works on this box before the encoder exists. **RUN 2026-07-06:
  MMAP-EXEC OK** — RW map + copy + mprotect(RX) + indirect call all
  permitted; f(2,3)=5 and f(2^64−1,2)=1 (silicon add IS wrap64). The
  harness pattern is the seed of the real differential runner.
- **Probe A — the model + fuel law**: minimal structured x86_64 model
  (GPR file with real names, SysV call semantics + clobber discipline,
  operand datatype, std/mem substrate, ~Ring-0 op subset); hand-play
  one straight-line piece and one loop piece through the standard cert
  form. Go/no-go = every §7 checklist item has an instantiation.
- **Probe B — encoder + silicon differential**: flattening encoder to
  real bytes + the harness replaying model-computed vectors (the
  wasm_diff vector grammar carries over); settles the trap-leg
  question.
