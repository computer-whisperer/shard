# x86_64 — the second target: scope and design ledger

Status: design ratified in discussion 2026-07-06; this file is the
record. The model instantiates the target-model contract of
docs/LOWERING.md §8; the front/back split (§6ae) is what makes a second
target a bounded exercise — tools/low/{shape,schema,lin,doc}.shard
import unchanged, proof.shard forks, a new back end plays wasmgen's
role. This document pins scope and the abstraction ledger BEFORE the
model exists; probe findings and slice records get appended as they
land.

## 1. Why x86_64, and why now

The wasm pilot is complete through Ring 0 + the loop-fragment family +
the front/back split; the §8 contract is distilled from everything it
needed. x86_64 is the target that tests the contract's genericity where
wasm was kind: no typed frames for free, no structured control in the
machine, flat shared memory.

Straight to x86_64 — **no 32-bit x86 model, ever**. The easiest-first
rung 32-bit would have occupied was wasm's explicit job, and 32-bit is
worse on every axis that matters here: cdecl passes args on the stack
(forcing the stack question immediately), 8 GPRs vs 16, and no native
differential without compat modes. x86_64 SysV passes the first six
args in registers, which composes directly with the frame design below.

The dev box IS the target: the engine-differential contract item (§8.6)
is satisfied by **real silicon** — a harness that writes encoded bytes
into an executable page and calls them with SysV register args (the
x86 analog of wasm_diff.mjs/V8, and the realest reality gate
available).

## 2. The governing principle: performance parity

User-set bar: the architecture must place no blocker to eventual parity
with C+gcc / Rust+LLVM workloads. Parity is not today's deliverable;
today's obligation is that **every abstraction in the model is
classified** into one of three bins:

- **permanent and perf-neutral** — abstractions matching the contract
  gcc/LLVM themselves offer their users; never removed;
- **training wheels with a named removal path** — v1 omissions whose
  later addition is grammar/convention GROWTH, touching no existing
  cert;
- **rework bombs** — decisions whose later migration would re-template
  the proof corpus. These get decided NOW, in favor of real x86_64
  usage.

One standing perf note in the pipeline's favor: PRE discipline
discharges bounds by proof, so safety costs zero runtime checks — the
certified pipeline is AHEAD of parity on that axis, not behind.

## 3. Permanent abstractions (perf-neutral)

1. **Structured (reducible) control.** The model executes a program
   VALUE — an AST with regions/loops/ifs — never bytes from memory.
   Compilers compile reducible CFGs; performance never comes from
   irreducible flow. The real perf patterns this seems to exclude all
   have structured representations proven by wasm itself and enter as
   grammar growth: jump tables (BrTable analog), indirect calls
   (call-through-table), tail calls.
2. **Code is not data (Harvard split in the model).** Mid-instruction
   jumps, overlapping decode, self-modifying code are INEXPRESSIBLE —
   excluded by the grammar, not policed by a checker. The flattening
   encoder (structured control → labels → relative jumps → bytes) is a
   differential-covered trust leaf exactly like wasm's encoder.
3. **Private control stack.** Return addresses and the call/ret
   mechanism stay model-level and are never addressable. Zero perf
   cost — no legitimate C/Rust pattern reads a return address (UB
   there too); LLVM offers the same guarantee shape.

## 4. Commit-now decisions (rework bombs if deferred)

1. **The real 16-GPR register file, in the model, day one.** Register
   allocation is the EMITTER's job at generation time, so the generated
   cert speaks about the ALLOCATED program. This exploits the
   pipeline's own LCF discipline: the generator is untrusted, the
   output is proof-checked, so allocation correctness is swallowed by
   the per-program proof — no verified-regalloc arc ever, a near-1:1
   encoder (small trust leaf), and performance-relevant placement
   visible in the certified layer. The rejected alternative (unbounded
   virtual slots, encoder allocates — the wasm-shaped comfortable
   choice) parks regalloc in the uncertified leaf AND re-templates
   every proof at the eventual migration. Proof templates spell
   rdi/rax instead of local indices; comparable complexity to wasm's
   locals arithmetic.
2. **SysV calling convention from day one.** Args in
   rdi/rsi/rdx/rcx/r8/r9, return in rax. The caller-/callee-saved
   split enters the standard cert form as a **register clobber set** —
   the register analog of memory footprints: the call-composition
   keystone carries "callee preserves these registers" the way
   footprint companions carry "callee doesn't touch these bytes."
   Starting wasm-style (calls preserve everything) would bake a
   fiction into every consumer proof.
3. **The control/data stack split.** Control stack private forever
   (§3.3). The DATA stack — spill slots, stack-allocated buffers,
   out-params — must eventually be ADDRESSABLE Mem, because heap-free
   stack allocation is the heart of the embedded-Rust patterns pinned
   as first-class. It is a convention, not a mechanism: a
   stack-pointer register + footprint discipline over a Mem region
   (the §7.3 footprint-companion promotion), added when a consumer
   forces it. LLVM→wasm makes exactly this split (protected call
   stack + shadow stack in linear memory) at near-native performance.
   The pin here is negative: frames are NOT unaddressable-forever.
4. **Value domain [0, 2^64), wrap64 collapse family.** The
   rep-collapse lemma family (§8.4) at modulus 2^64. Engine
   consequence probed before the model (probe 0a below): 64-bit
   constants and 2^128-scale ground products exercise the compiled
   chain's BigInt path where the wasm pilot never left i63 territory.

## 5. Training wheels (named removal paths, all additive)

- **Byte-width memory ops only** → wide loads/stores (mov
  byte/word/dword/qword). std/mem's `mem_read` width-first surface
  anticipates this; growth = new ops + collapse lemmas, no existing
  cert touched.
- **Register-only operands** → real addressing modes. The operand
  position in the instruction grammar is a DATATYPE from day one; v1
  populates only the register arm; [base + scale·index + disp] arrives
  as new arms + emitter selection, not a grammar rework.
- **FLAGS stay out of the model.** Comparisons fuse into structured
  conditions (the If carries the comparison; encoder emits cmp+jcc) —
  the cond_op shape. Named perf exceptions enter later as COMPOUND ops
  when consumers arrive: x86 div yielding (quot, rem) in one
  instruction, widening 64×64→128 mul, adc for bignum chains.
- **One canonical encoding per instruction.** The encoder is total on
  the grammar, mechanical REX+ModRM, zero optimization. Encoding
  selection, alignment, scheduling, uop fusion are semantics-neutral
  encoder/micro-arch concerns — differential-covered, never certified.
- **No spills in v1** (current fragments fit in 16 registers); spill
  vocabulary = data-stack slots per §4.3 when something forces it.
- **Optimization generally**: the pipeline is already on the
  translation-validation side of the CompCert tradeoff — inlining is
  ratified as "a general certified transform" and is the template for
  later scheduling-class rewrites.

## 6. SIMD (eventually; extension points only, nothing built)

Machine state carries register CLASSES from day one: the GPR file now,
an XMM/YMM file absent-or-empty until used — the precaution is that
state shape must not bake in "registers = one Int list." Op tables are
the extension mechanism; the wasm lesson is pinned: the Bop vocabulary
is FIVE tables wide (semantics / encoder / cert readback / render /
emitter conditions) — extend in step or drift. Lane representation
(List Int vs wide Int) genuinely deferred.

## 7. The §8 contract instantiation checklist

What "add x86_64" means, item by item (LOWERING.md §8):

1. Fuel big-step, additive slack: same discipline; call entry burns
   fuel; loops re-enter at fuel−1 (structured control makes the wasm
   law transplant directly — probe A verifies).
2. Named SCC stop points: eval_call/eval_loop/eval_seq analogs in the
   x86 evaluator's recursion split.
3. Call-composition keystone: call_bridge analog, EXTENDED with the
   register clobber set (§4.2) — the one place the x86 keystone is
   richer than wasm's.
4. Rep-collapse family: wrap64_id + friends (probe 0a stresses the
   constants).
5. Literal-spelling discipline: unchanged (ENC opacity law applies to
   any Nat/register-name enc fns riding into machine state).
6. Encoder + engine differential: flattening encoder + the
   mmap-and-call silicon harness (probe 0b proves the leg exists).
   Trap leg: where the model says None, the harness must observe a
   signal (SIGFPE/SIGSEGV) or refuse-by-construction — open question,
   settle at probe B.
7. Memory denotation: std/mem's observational substrate UNCHANGED —
   same Mem, same mask-on-read, same framed arbitrary-m0 statements.
   Flat shared memory means footprint companions (§7.3) promote from
   QoL toward mandatory as multi-module x86 builds arrive; let a real
   consumer force the timing.

## 10. The call arc (landed 2026-07-06) — §7 item 3

The model grew `xrun_regs` — the regs-level denotation returning the
FULL final register file plus memory (`xrun_fn` is now its rax
projection) — and `xcall_bridge`, the call-composition keystone: a
folded in-body `(xeval_call (S f2) …)` term (what a stopped compute
leaves at a call site) completes exactly as `xrun_regs` of the callee
on the CALLER'S registers and memory. Premise 0 is ground bookkeeping
(`xfunc_at` computes at any concrete site); premise 1 is the callee's
run — the slot the callee's own conformance cert discharges. The proof
is SMALLER than wasm's `call_bridge`: `xeval_call` does no
pop/reverse/zero-extend argument marshalling, the register file passes
through whole.

**The register clobber set rides in the cert FORM, not in any new
mechanism.** A callee cert states `xrun_regs` with every register
field quantified — the entry file spelled as a `MkRegs` constructor
literal over fifteen symbolic fields, the `xlgw_fill` idiom. (A
`with_`-spine on a VARIABLE record would be stuck: record matches
reduce only on constructor literals. Field-quantification is what
keeps the cert pure compute.) The cert's result record literal then
reads off as the clobber set: changed fields are the clobbers, fields
passed through verbatim are preserved — the register analog of the
framed arbitrary-m0 memory schema (§4.2), and the thing the emitter
back end will generate per callee.

Hand-play (examples/x86_pieces.shard stage 3; everything first check):
`xsq_cert` — clobber set = {rax}, fourteen fields verbatim — welded
TWICE into the two-call consumer `xsumsq_thm` through `xcall_bridge`.
The consumer parks a live value in rcx ACROSS the second call,
justified purely by rcx passing through the callee's cert, and bumps
rdi after the first call for the same reason; the callee's body is
never computed into. Fuel is exact S-tower + slack; the bridge's
`(S f2)` pattern binds against the concrete tower and the cert's slack
variable absorbs the difference (the additive-slack discipline doing
its job at a composition boundary).

Named growth remaining (fences, not blockers): the encoder still
lowers XCall→int3 — inter-function LAYOUT (function offsets in a flat
image, call rel32) belongs to the emitter back end; memory-effect
callee certs compose through the same premise slot when a consumer
arrives; callee-saved discipline for generated code (which registers
an emitted callee may use at all) is emitter policy, checked per-piece
by the cert form above.

## 12. The emitter back end, slice 2: the if fragment (2026-07-06)

**Tail-position ifs, the §5 fused-condition design realized.** The
model grew relational `Cond` arms (CEq/CLtU/CLeU, a dst register vs a
Src — the register analog of wasm's BEq/BLtU/BLeU cond_op) and the
encoder grew cmp+jcc (0x39 /r reg-reg, 0x81 /7 imm32; je/jb/jbe TAKEN
when the condition holds), with `xsz_brif` making the branch width
computable per (op, operand shape) so the flattening's fixed-width
invariant survives (reg cmp / test = 9 bytes, imm32 cmp = 13). This is
the one genuine model+encoder growth of the slice; it is additive
(existing CEqz/CNz loop guards untouched, all pieces re-verify) and
proven on silicon (x86_diff_run.shard grew gate/le/cmpreg/eqsel
diamonds — the CPU agrees 26/26, cmp/jcc included).

**The diamond is SMALLER than wasm's.** x86 keeps structured control
(XBlock/XBrIf/XBr), so the emitter reuses the wasm diamond SHAPE —
`XBlock[ XBlock[ XBrIf COND 0, ELSE.., XBr 1 ], THEN.. ]` — but the
comparison is FUSED into the XBrIf, so there is no stack value and no
separate cond code (wasm computes b2i on the stack; x86 doesn't).
tools/low/shape.shard's RT region-tree vocabulary imports UNCHANGED
(the §6ae front end serving a second back end at the branch layer, not
just the leaf).

**Register allocation across branches — the parked-binding case.**
Both arms of an if deposit at the SAME result register rl (the whole
tail spine funnels to the function result, so rl is stable down the
tree); arms use scratch above rl, lets bind below the scratch window
and SURVIVE the branch. When a let precedes an if, the emitter reserves
rax for the result and binds below it (the straight-line path keeps rax
as first scratch — the two paths diverge cleanly, detected by
`has_tail_if`, so straight-line output stays byte-identical). selsq
(`let s = x*x in (if y=0 then s else s+y)`) is the pin: s parks in r10,
read by both arms, the if result lands in rax, no clobber.

**PROOF: the region tree drives a case-on per if.** After compute lhs
stalls the machine at the fused XBrIf (the test computes to exactly the
source condition — ground wrap collapse, no b2i intermediate), the
proof case-ons the source condition; each arm rewrites the ctor-named
hyp into both sides and computes past the branch, then discharges its
own wrap nodes (deduped against ancestor regions) with the wrap64_id
chain. Pre-branch let nodes discharge before the case-on; nested ifs
nest case-ons (clamp2). Premises are GLOBAL here (every wrap node a
plain premise) — the §6j condition-relative hygiene (arm-local
premises stripped in-arm, path-derivable bounds discharged silently) is
the named next hygiene slice. Fuel is over-provisioned (2·count+10);
additive slack absorbs the excess.

**enc_elf (models/x86/elf.shard) — untrusted container plumbing.** One

## 21. The bin rung opens — the harness dissolves (2026-07-08)

**The reshape (user-ratified 2026-07-08).** §20's "next" — a CLI runner
whose generated stub parses argv and guards the domain — was still
harness-shaped: program logic living in untrusted plumbing. The
governing intent, reinforced: **shard is a C-class compiled language —
the harness and interpreter systems dissolve COMPLETELY, leaving bare
hardware execution. Most of the system's complexity exists to ALLOW
that clean dissolution.** (This cuts against the functional-language
groove of baking a runtime into the language; the proof apparatus is
entirely build-time, and the runtime residue of the whole system is
proven bytes + crt0-sized glue + the syscall shims — no interpreter, no
GC, no runtime dispatch.) The rung is therefore `(bin …)` lowering to a
PLAINLY EXECUTABLE ELF, gcc-shaped: main = proven shard code (v1: pure
one-shot — argv bytes in, output bytes + exit code out), `_start` glue
= byte-MOVING plumbing under a signed contract (lay argv into memory,
call main, sys_write the output, sys_exit the code). The law: plumbing
MOVES bytes, proven code INTERPRETS them — parsing, validation, and
rendering are program logic, so decimal atoi/itoa live in the proven
image (std/str territory), not the stub. At the bin boundary the
accepts surface drives to EMPTY: a program's main is total over raw
input bytes; validation is the program's job, not a guard's (the
FFI-guard story stays with `(lib …)`, which keeps its cdylib role and
§20's self-checking ELF as ITS engine gate). Example-demanded growth
pulled forward: the writable second PT_LOAD (argv in memory), x86 div
(decimal itoa needs div/mod 10), parse loops over the mem fragment. A
pure one-shot main defers the World-effect axiom exactly per the §20
ruling — interactive I/O is the rung AFTER, where the axiom, the shim
citations, and the percolation-twin gate (image shims == declared
externs) land.

**The de-risk probe (landed, examples/binelf_probe.{shard,sh}).**
Hand-driven two-segment ELF confirming every new mechanism before
pipeline wiring: (1) a second RW PT_LOAD at 0x600000 as pure BSS
(p_filesz=0, p_memsz=4096 — the kernel zero-fill is load-bearing: the
glue stores only the low length byte into an already-zero 8-byte LE len
field); (2) the entry-stack contract ([rsp]=argc, argv pointers follow,
strings NUL-terminated; argc<2 → sys_exit(1), the error leg
gate-checked); (3) the glue copy loop — argv[1] bytes to NUL, MAXLEN
256 guard — into `[len:8][payload]` at the segment base; (4) an image
function reading the buffer per the mem-fragment convention and
returning `(len<<32)|bytesum`; (5) glue sys_writes the 8 result bytes,
exits 0. **The convention finding that shapes the glue contract: mem
exports take ABSOLUTE process addresses in the SysV arg registers (the
pointer in rdi IS the address — no separate base register), the same
marshalling the §20 stub already does.** The encoder grew three
additive glue constructors (mov r64,[rsp+disp8]; bare jcc/jmp rel32 —

constructors (XLoad8/XStore8 &c). Both existing x86 gate scripts re-ran

## 23. The memory WINDOW — A1's prerequisite (2026-07-08)

**A prefix is a wasm-ism; a process has a window.** `XModule` carried a
single `xmemsize`, and both memory guards read `0 <= p AND p + 1 <=
xmemsize` — memory as the PREFIX `[0, memsize)`. That is faithful for
wasm, whose linear memory really is addressed from zero. It is never
faithful for a Linux process, because page 0 is unmappable. The record
is now `(record XModule (xfuncs …) (xmemlo Int) (xmemhi Int))` and both
guards read `xmemlo <= p AND p + 1 <= xmemhi`: the half-open WINDOW the
process actually maps.

**Why this is soundness, not tidiness.** Under the prefix guard the
model declared a load at address 5 VALID (`0 <= 5`, `6 <= memsize`)
while silicon SIGSEGVs. Measured, before the change, against the
differential harness at the vectors' own `memsize = 0x40010000`: the
DATA_BASE load agreed, the address-5 load came back `[hardware
faulted]`. So a lowering cert — whose whole content is
`xcall_fn_mem … = Some (Pair …)` — could be PROVEN for a binary that
faults. Premises cannot patch this: a premise constrains the entry's
ARGUMENT, not what the code does with it, so `BUF <= x0` still admits
code that loads at `x0 - 1000000`. Only the guard can refuse it. With
the window, a stray access traps, a trap is `None`, and the `Some (Pair
…)` conclusion becomes UNPROVABLE. "The code never touches unmapped
memory" stops being a hope and becomes a CONSEQUENCE of the cert.

**Why A1 forced it.** Nothing was unsound before: no cert'd mem module
was ever executed as a process. The mem certs live at `memsize = 65536`;
the silicon differential runs SEPARATE vector modules at DATA_BASE and
tests the encoder, not a cert instantiation — the §15 address
dispensation, parked. The bin rung is the first artifact where the cert
and the silicon run are the SAME module (IMGTIE ties them). A1's entry
reads its payload at `BUF = 0x600000`, so the cert must hold at
`x0 = BUF`, so the window must contain BUF. The gap goes live exactly
here, which is why the window lands as A1's prerequisite rather than as
part of it.

**Where each consumer now states its truth.** Existing certs: `0 65536`
(lo = 0, so every premise's text is UNCHANGED — `xmemlo_of` reduces
under `compute` precisely where the `0` literal used to sit, and not one
proof step moved). Pure-register modules: `0 0`, the empty window,
admitting no access at all. The differential vectors: `1073741824
1073807360` = `[DATA_BASE, DATA_BASE + 0x10000)`, which finally makes
x86_diff.c's `MAP_FIXED` comment TRUE — the model now says exactly what
the harness maps, and an address below DATA_BASE traps as it should.
`std/mem/mem.x86.shard`'s callee certs quantify `mlo`/`mhi` where they
quantified `msz`, so a bin's window cites them unchanged.

**The low bound gets SILICON coverage, not just model coverage.** Every
pre-existing vector sat at or above DATA_BASE — the lowest landed
*exactly* on the new `xmemlo` — so the new lower bound would have shipped
with no differential leg at all. x86_diff_run's `put` vector at
`0x3FFFFFFF` (DATA_BASE − 1, the last byte of the preceding page) closes
it: the model traps, the CPU SIGSEGVs, the harness scores the caught
signal as agreement (34/34). It is a real regression guard, demonstrated
by mutation — restore the prefix lower bound on the store guard and the
model predicts a SUCCESSFUL store (`-> 41`) where the CPU faults, so the
differential goes red with `1 disagree`. That is precisely the original
bug, now caught by an automated gate rather than by a hand probe.
x86_diff.c itself is UNTOUCHED.

**The fixture (examples/x86_window_law.shard, corpus-pinned).** Ten
machine claims over `[100, 200)`, FOUR LEGS ON EACH GUARD — `XLoad8` and
`XStore8` carry independent copies of the bound, so a store-side
off-by-one is invisible to every load claim. Per guard: in-window
succeeds; the last byte 199 succeeds and 200 traps (the half-open
boundary); below-window 50 and address 0 trap. The below-window leg is
the one a prefix guard could not even STATE — it is the fixture that
would have caught the gap.

MUTATION-VERIFIED, not asserted. Each bound of each guard was mutated in
turn, and exactly the claims for that guard's that end go red: prefix
lower bound on the load guard → `window_below_traps` + `window_zero_
traps`; on the store guard → their store twins; `(le p hi)` for
`(le (+ p 1) hi)` on the store guard → `window_store_hi_traps` alone.
(Scoped to this file's own `window_*` claims: `std/mem`'s `xmget_cert`
also reddens under a load mutation, since the fixture imports it.) The
first cut of this fixture had only two store legs and the store's
high-bound mutant escaped every one of its claims — caught in review,
not by the fixture, which is the argument for the four-leg shape.

NON-VACUITY, and it is not free: `xrun_regs` collapses fuel-exhaustion to
`None` just as it does `XTrap`, so a bare `= None` claim could be
satisfied by running out of fuel rather than by trapping. The in-window
claims run the SAME fuel `(S (S Z))` on the same modules and return
`Some` — which is what makes the trap claims mean trap.

**Blast radius, in the event.** Two guard sites; `xmemsize` no longer
exists anywhere in the repo. Eight generated cert files regenerated for
free through x86gen + shardfmt; the emitter's `65536` module literal
became `0 65536` (premise text untouched — parametrizing the window on
`(lo, hi)` is A1's job, and is the same move as the parametric `WrapK`
hygiene pass). bytetie's and lowcheck's module-slot reflectors read two
literals, and BOTH must be `SInt` — a non-literal `lo` now refuses
loudly, as its wasm twin always did for memsize.

**The wasm/x86 memory datum is a SUM, by construction.** bytetie's
`LSlot` carried the module's memory size as a bare `Int`. Handing it the
window's `hi` would have been inert today (the x86 lib tie never reads

silently receive `hi` under the name `memsize` — the type won't let it.

All 16 lowbuild builds green, silicon differential 34/34, sweep clean,

## 26. A2 slice 1: the copy loop — the entry WRITES what it read (2026-07-08)

The store value `(mem_get m ADV-ACCUM)` — meta/shape's `LVRead`, which
`lp_recognize` has always produced and wasmgen has always emitted, and
which x86gen refused. A2's shape.

**The machine delta is forced by the ISA.** x86 has no mem-to-mem move,
so the byte lands in a register: `XLoad8 RAX [rdi]` / `XStore8 [rsi]
RAX`. That drags in the §24 Int-return probe's whole rax apparatus —
quantify the entry rax, close the exit rax over a residue fn — on top of
mark's two-advancing-accum Mem-return worker. Two address guards per
iteration (the load, then the store), so the guard stage runs twice. The
piece's epilogue `XMovRI RAX 0` overwrites the residue, so the
Mem-return conclusion keeps its `(Pair 0 mem')` shape. 7-instruction
body → S^11/S^15 (the instrs+4 rule).

**The one non-mechanical delta — and it was a real bug waiting.** The
residue CANNOT be `xlg_last`, which reads a FIXED memory: faithful for
the read-only sum loop, wrong here, where each iteration's store changes
what the next load sees. `xlg_lastc` threads the memory exactly as the
spec's own recursion does, so overlapping src/dst stays faithful for
free. This is not a proof-engineering nicety. The differential now
carries an OVERLAPPING forward copy (`dst = src+1`, seed `ABC`): the
threaded semantics say the loop smears byte 0 to `AAAA`; the naive
non-threading answer is `ABCA`. **Silicon says `AAAA`.** A fixed-memory
residue would not merely have failed to prove — it would have been
WRONG about the hardware, and the differential now pins that.

**The vector form grew (`PXVS`, the XSEEDCASE wire).** `PXV` documented
"no initial image — write-only loops need none." The copy loop is the
first loop that READS what it stores, so that reason expired: on a
zeroed page it copies zeros to zeros and its data path never reaches
silicon. `PXVS` adds a seed image at an absolute address, written after
the page is zeroed and before the call, so the readback tests the bytes
the loop actually moved. Additive: `XMEMCASE`/`XCASE` untouched, wasm
builds refuse `PXVS` as they refuse `PXV`. (Negative controls: removing
the seed, and expecting the naive overlap answer, both make the
differential disagree; an out-of-page seed refuses.)

**v1 fences (loud, named growth):** exactly two accums, both `+1`, the
READ address first, the store address second — the probe's pinning. A
reversed-role or three-accum copy refuses by name; a static destination
refuses at the store classifier. Probe:
examples/xcopyloop_probe.shard (both claims first generation; four
mutations refuse). x86loop ships `xl_copy`; 8/8 silicon.

## 28. A2 slice 2b: the transform loop — the machine branches on a byte (2026-07-08)

**A diamond inside a loop body.** The ISA has NO cmov and NO setcc —

## 32. The World-effect rung: kernel model ratified (2026-07-09)

**RATIFIED (user, 2026-07-09, three-turn design pass).** The trust
locus for effects is a WRITTEN-IN-SHARD KERNEL MODEL, not per-shim
axioms. "Linux does what our model says" gets exactly the epistemics
"silicon does what models/x86/x86.shard says" already has: a
definitional transition function (no in-language axiom — the models
tree stays theorem-only), empirical differential gates, and a new
ledger trust kind beside the x86 one. Consequence, SUPERSEDING the
platform-externs ruling's "axiom-pedigree shim cert" text and the
TRUST STATUS header of models/x86/linux.shard: the shim bodies (mov
args, `syscall`, ret) become ORDINARY PROVABLE CODE against the
extended model, their §10-form callee certs THEOREM-pedigree. The
trusted surface contracts to one reviewed model file + the gates. The
only non-shard content in the tower is the model and two bytes of
encoding.

**Effect-points, not instructions (the Windows stress-test).** Windows
has no stable syscall ABI (numbers churn per build; the stable
boundary is ntdll/kernel32 exports), so the cert form, trace
vocabulary, and gates speak EFFECT-POINTS — per-platform: on Linux the
`syscall` instruction with id = rax; on Windows (later) a call through
an IAT slot. The platform-API surface speaks platform-neutral events
(wrote fd bytes), never registers.

**The model fragment (a chosen discipline, like totality).** v1 covers
synchronous single-threaded programs: effects happen ONLY at
effect-points — that is the axiom that keeps the tower purely
computational between stops. Asynchronous kernel mutation (signals at
arbitrary instruction boundaries, shared-VM clone, io_uring rings) is
OUT OF THE FORM, permanently; Linux's own sync alternatives (signalfd,
epoll, multi-process + pipes) are the sanctioned path. mmap/brk
(window becomes a SET of windows — machine-model growth) and fork
(process-tree traces) are NAMED growth rungs. ioctl is incremental:
each used ioctl gets a model arm, unmodeled = refused.

**The no-replay gate rule.** The model is never in the execution path;
proofs scale with program TEXT (one inductive trace invariant per
loop), and model-fidelity gates are per-syscall vectors plus small
end-to-end pins. STANDING RULE: no gate may require whole-execution
model replay.

**Nondeterminism = world-as-oracle.** Syscall returns are READ FROM
the world value; events are recorded to a trace; determinism is
restored by making nondeterminism an input, and specs quantify over
worlds. The `syscall` instruction's rcx/r11 clobbers (rip/rflags,
per linux.shard's ledger) are also oracle-fed — faithful-by-
nondeterminism, so no cert can depend on them.

**The four layers.** (1) kernel model, per-platform, the trust locus;
(2) shims, per-arch, PROVEN, ~5 instructions each; (3) portable proven
shard — write_all retry loops, errno decoding; (4) the friendly
surface: an opaque directory module whose laws are stated over traces,
per-platform backends discharging them. Process startup (argc/argv
stack layout) moves into the model at a later rung, retiring the last
unchecked glue.

**Implementation shape (proposed at ratification; details below are
the working plan).** Grounding against the code killed the design
discussion's "stop outcome" sketch: XOut has no continuation
structure, so a syscall stop cannot resume through xeval_seq/loop/call
without a CPS rework of the whole evaluator. Instead: a WORLD-LAYER
EVALUATOR FAMILY (`weval_*`, models/x86/world.shard) mirroring the
base family's recursion, threading a World, whose XSyscall arm applies
the kernel transition and whose XCall arm DELEGATES: callee without
effect-points (static predicate `xfn_eff`) => cite xrun_regs BY
DEFINITION — so every existing pure cert plugs into world-layer proofs
with no bridge lemma; effectful callee => recurse in-world. New
terminal outcome `WExit` (sys_exit stops execution mid-function and
percolates up like XTrap). The base evaluator's XSyscall arm is
`(Some XTrap)` — effect-points are outside the pure fragment, and
None stays exclusively fuel. A bin's whole behavior becomes ONE
equation: `xrun_w … = Some (WExit trace)` with the trace naming every
effect; BINSURFACE speaks the trace. v1 restriction: effect-points
only in straight-line spine positions (no syscall-in-loop; write_all
